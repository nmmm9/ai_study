왜 하나의 검색 방법으로는 부족한가?

Vector Search만 쓸 때 실패

질문: "LH 청년 매입임대주택"
→ 고유명사 "LH" "매입임대"를
벡터가 일반화해버림
→ 관련 없는 문서가 상위에 올 수 있음

BM25만 쓸 때 실패

질문: "집 구하기 힘든 청년 지원"
→ "집" "구하기" 단어가 문서에 없으면
"주거" "임차" 와 매칭 실패
→ 의미는 같지만 검색 안됨

항목 BM25 (키워드) Vector (의미)

API 필요 없음 (로컬 계산) 임베딩 API 필요
4 BM25 원리
Best Match 25 | TF-IDF의 개선된 버전

BM25 공식 직관적 이해

BM25 점수 공식
score(d, q) = Σ IDF(q_i) × [ TF(q_i, d) × (k1 + 1) ] / [ TF(q_i, d) + k1 × (1 - b + b × |d| / avgdl) ]

IDF (희귀성)

전체 문서 중 이 단어가 있는 문서가 적을수록 점수 높음.
"청년"은 흔하니 낮음. "LH공사"는 희귀하니 높음.

TF (빈도) + k1

단어가 많이 나올수록 점수 높지만 k1로 상한 제한.
같은 단어 100번 반복해도 의미 없이 점수 오르지 않음.

문서 길이 보정 (b)

긴 문서는 단어가 많아서 자연히 점수 높음.
b로 길이 차이를 보정해 공평하게 비교.

내 코드에서 BM25가 작동하는 방식

# vector_store.py def _tokenize(text): # 한국어: 공백·특수문자로 분리, 소문자 통일 return re.findall(r'\w+', text.lower()) # "청년 주거 지원" → ["청년", "주거", "지원"] def _rebuild_bm25(self): # 문서 추가/삭제할 때마다 인덱스 재구성 tokenized = [_tokenize(doc) for doc in self._bm25_docs] self._bm25 = BM25Okapi(tokenized) # ChromaDB와 BM25 인덱스를 항상 동기화
⚠️ 한국어 BM25의 한계: 형태소 분석 없이 단순 공백 분리를 사용합니다. "주거지원"과 "주거 지원"을 다른 단어로 인식할 수 있습니다. 더 정확하게 하려면 KoNLPy(Mecab) 같은 형태소 분석기를 사용해야 합니다.
5 RRF (Reciprocal Rank Fusion)
두 검색 결과를 점수가 아닌 순위로 융합하는 알고리즘

왜 점수를 그냥 더하면 안 되나?

BM25 점수(예: 15.3, 8.2...)와 Vector 유사도(예: 0.87, 0.72...)는 단위와 스케일이 완전히 다릅니다. 그냥 더하면 BM25 점수가 압도적으로 큽니다.

RRF는 두 결과의 순위(1위, 2위, 3위...)만 사용해서 공정하게 합칩니다.

RRF 공식
RRF_score = vector_weight × 1/(k + vector_rank) + bm25_weight × 1/(k + bm25_rank)
k=60 (순위 차이 완화), vector_weight=0.7, bm25_weight=0.3

RRF 시각적 예시

Vector 순위
1위청년 전세 보증금0.91
2위LH 매입임대0.82
3위청년 월세 지원0.79
4위주거급여 신청0.71
BM25 순위
1위LH 매입임대18.4
2위청년 월세 지원14.2
3위청년 전세 보증금11.8
4위주거급여 신청9.1
+
RRF 최종 순위
1위청년 전세 보증금0.7×1/61 + 0.3×1/63
2위청년 월세 지원0.7×1/63 + 0.3×1/62
3위LH 매입임대0.7×1/62 + 0.3×1/61
4위주거급여 신청0.7×1/64 + 0.3×1/64

두 검색에서 모두 높은 순위에 있는 문서가 최종적으로 높은 점수를 받습니다. "청년 전세 보증금"은 Vector 1위 + BM25 3위 → RRF 1위.

6 GPT Re-ranking
reranker_service.py | 검색 결과를 LLM이 다시 채점
현재 사용 Cross-Encoder (GPT)
  • 질문 + 문서를 함께 LLM에 입력
  • 두 텍스트의 관계를 직접 이해
  • 0-10점 채점 후 재정렬
  • 정확도 높지만 API 비용 발생
대안 Bi-Encoder (임베딩)
  • 질문과 문서를 각각 임베딩
  • 코사인 유사도로 점수 계산
  • = 벡터 검색과 동일한 방식
  • 빠르고 저렴하지만 정확도 낮음

Cross-Encoder가 더 정확한 이유

Bi-Encoder 한계:

질문: "LH 임대주택 신청 조건" → 벡터A 문서: "청년 주거 지원 방법" → 벡터B # 두 벡터를 독립적으로 생성 # 서로를 "보지 않고" 만든 벡터끼리 비교 # 맥락 이해 부족

Cross-Encoder 강점:

입력: "질문: LH 임대주택 신청 조건 문서: 청년 주거 지원 방법..." # 질문과 문서를 함께 보고 관계 분석 # Attention이 두 텍스트 사이에 작동 # "이 문서가 이 질문에 얼마나 답이 되나?" 출력: 7.5 / 10

왜 필요한가? — LLM의 "Lost in the Middle" 문제

LLM은 긴 컨텍스트에서 중간 부분의 정보를 잘 잊어버리는 경향이 있습니다. 관련 없는 내용을 걷어내면 LLM이 핵심에 집중합니다.

LLM 압축
↓
압축 후
→
핵심만 (50자): "소득 6분위 이하, 보증금 5천만원 한도"
현재 사용 LLM 기반 압축
  • 질문을 이해하고 관련 내용 추출
  • 의미적 압축 (문장 재구성 가능)
  • 200자 미만 청크는 건너뜀
  • "없음" 반환 시 해당 청크 제거
  • 가장 비쌈 (청크당 API 1회)
대안 키워드 기반 압축
  • TF-IDF로 중요 문장만 추출
  • 원문 문장 그대로 유지
  • API 비용 없음, 매우 빠름
  • 의미 이해 없이 단순 선택
  • 품질은 LLM 방식보다 낮음
