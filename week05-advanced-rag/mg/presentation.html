RAG = 검색 + 생성

LLM한테 문서를 주고 답변하게 하는 것

질문
→
문서에서 관련 부분 검색

검색 결과 기반으로 LLM이 답변

실제 문서를 찾아서 그 내용 기반으로 답하면 → 정확한 답변.

그런데... 검색을 얼마나 잘 하느냐에 따라 답변 품질이 완전히 달라진다.

Basic RAG의 한계

1

의미 격차

질문: "~란 무엇인가?" (의문문)
문서: "~는 ~이다" (서술문)

같은 내용인데 표현이 달라서 매칭이 안 됨

2

검색 순위 한계

코사인 유사도만으로 순위를 매기면
실제로 관련 없는 문서가
상위에 올 수 있음

3

검색 범위 한계

상위 5개만 가져오면
6번째에 있는 좋은 문서를
놓칠 수 있음

이 3가지 문제를 각각 다른 방법으로 해결하는 게 Advanced RAG

9가지 파이프라인

01

Basic RAG

질문 임베딩
질문 → 벡터
→
벡터 검색
유사한 문서 Top-5
→
LLM 답변
검색된 문서 기반

장점

가장 빠르고 저렴
구현이 단순
대부분의 간단한 질문에 충분

한계

질문과 문서의 표현이 다르면 놓침
코사인 유사도 순서에만 의존
상위 5개 밖의 좋은 문서를 놓칠 수 있음

02

HyDE — 가상 문서 임베딩

질문 대신 LLM이 만든 가상의 답변을 임베딩해서 검색

LLM이 가상 답변 생성
→
가상 답변 임베딩
→
검색
→
답변
왜 효과적인가?
질문: "RAG란 무엇인가?" → 의문문 벡터
가상 답변: "RAG는 검색 증강 생성으로, 외부 문서를 검색하여..." → 서술문 벡터
문서도 서술문이니까 → 서술문끼리 매칭이 더 잘 됨
03

Reranking — LLM이 검색 결과 재정렬

넓게 검색(Top-20)한 뒤, LLM이 실제 관련성을 점수(0~10)로 평가해서 재정렬

임베딩
→
Top-20 검색
넓게 가져오기
→
LLM 리랭킹
각 문서에 0~10점
→
Top-5 선택
→
답변
핵심: 코사인 유사도는 "벡터 거리"만 보지만, LLM은 문서를 읽고 이해해서 판단한다.
주의: 처음 20개에 포함 안 된 문서는 살릴 수 없음 → 넓게 검색하는 게 중요
04

Hybrid Search — 벡터 + 키워드 동시 검색

벡터 검색 (의미 기반)

"RAG 파이프라인" 검색 →
"검색 증강 생성 구조"도 찾아줌

의미가 비슷하면 찾음
하지만 "ChromaDB" 같은 고유명사는 놓칠 수 있음

BM25 검색 (키워드 기반)

"ChromaDB" 검색 →
정확히 그 단어가 있는 문서만 찾음

단어가 정확히 일치해야 찾음
"벡터 데이터베이스"로는 못 찾음

Hybrid = 둘 다 검색하고 RRF로 병합
의미도 맞고 + 키워드도 맞는 문서가 상위로 올라옴

05

Multi-Query — 질문을 다양하게 바꿔서 검색

LLM이 원래 질문을 3가지 다른 표현으로 변형 → 각각 검색 → 합산

원래 질문: "RAG에서 검색 품질을 높이는 방법은?"

LLM이 만든 변형

1. "RAG 파이프라인의 retrieval 성능을 개선하는 기법은?"

2. "벡터 검색 결과의 정확도를 높이려면?"

3. "LLM 기반 질의응답에서 문서 검색 최적화 방법은?"

4번 검색 → 중복 제거 → 다양한 표현으로 작성된 문서까지 발견

06

Self-RAG — 스스로 평가하고 교정

답변을 생성한 뒤 LLM이 스스로 품질을 평가하고, 부족하면 재생성

검색 필요?
LLM이 판단
→
검색+생성
→
자가 평가
점수 0~10
→
6점 이상 → 통과

평가 기준 1

답변이 검색된 문서에
근거하고 있는가?

평가 기준 2

질문에 정확히 답하고 있는가?
빗나간 답변은 아닌가?

6점 미만이면?

피드백을 반영해서
답변을 재생성

07

CRAG — 검색 결과가 별로면 다시 찾기

Self-RAG가 답변을 평가한다면, CRAG는 검색된 문서 자체를 평가

CORRECT

검색된 문서가
질문에 답하기 충분함
→ 바로 답변 생성

AMBIGUOUS

부분적으로 관련 있지만
충분하지 않음
→ 쿼리 수정 후 재검색

INCORRECT

검색된 문서가
질문과 거의 관련 없음
→ 완전히 다른 쿼리로 재검색

Self-RAG vs CRAG 차이:
Self-RAG: "내 답변이 맞나?" → 답변을 평가
CRAG: "내가 찾은 문서가 맞나?" → 검색 결과를 평가
08

Adaptive RAG — 질문 난이도별 자동 선택

LLM이 질문의 복잡도를 판단해서 적절한 파이프라인을 자동으로 선택

복잡도질문 예시선택되는 파이프라인
SIMPLE "임베딩이란?" Basic RAG — 빠르고 저렴하면 충분
MODERATE "HyDE와 Reranking의 차이?" Rerank RAG — 비교 질문은 정밀도 필요
COMPLEX "대규모 문서에서 검색 품질과 비용을 모두 최적화하려면?" Advanced RAG — 복잡한 추론 필요

핵심: 간단한 질문에 비싼 파이프라인을 쓸 필요 없다 → 비용/시간 최적화

전체 비교

기법속도비용품질해결하는 문제
Basic빠름저렴보통기준선 — 단순 질문에 충분
HyDE보통보통높음질문↔문서 표현 방식 차이
Rerank보통보통높음검색 순위의 부정확성
Advanced느림비쌈최고의미 격차 + 순위 문제 동시 해결
Hybrid빠름저렴높음키워드/고유명사 누락
Multi-Query느림보통높음단일 질문의 검색 한계
Self-RAG느림비쌈최고할루시네이션 방지
CRAG느림비쌈최고검색 실패 대응
Adaptive상황별상황별상황별비용/시간 최적화

언제 뭘 쓸까?

"검색은 되는데 엉뚱한 게 나와"

→ HyDE: 질문과 문서의 표현이 다를 때
→ Rerank: 순서가 이상할 때
→ Advanced: 둘 다일 때

"좋은 문서가 아예 안 나와"

→ Hybrid: 특정 용어/이름을 놓칠 때
→ Multi-Query: 질문 표현이 제한적일 때
→ CRAG: 검색 자체가 실패할 때

"답변을 신뢰할 수 없어"

→ Self-RAG: AI가 스스로 답변 품질을 평가+교정

"매번 어떤 기법 쓸지 고르기 귀찮아"

→ Adaptive: AI가 질문 난이도 판단해서 자동 선택

핵심 정리

1. 검색 품질 = 답변 품질

LLM이 아무리 똑똑해도
잘못된 문서를 주면 잘못 답한다
좋은 검색이 좋은 답변을 만든다

2. 만능 기법은 없다

빠름 ↔ 정확, 저렴 ↔ 고품질
항상 Trade-off가 존재
상황에 맞게 선택하는 게 핵심

3. AI가 스스로 판단하기 시작

Basic~Rerank: 시키는 대로 실행
Self-RAG/CRAG: 스스로 판단하고 교정
→ Agentic RAG로 진화하는 중

4. 기법은 레고처럼 조합 가능

HyDE + Rerank = Advanced
Hybrid + Self-RAG도 가능
필요한 블록만 골라서 조립
