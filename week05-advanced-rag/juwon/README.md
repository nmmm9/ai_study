## 핵심 구현

### 🔹 주요 로직 설명:

본 시스템은 기존 RAG 구조를 확장하여 검색 성능을 개선한 Advanced RAG Pipeline으로 구성하였다.

전체 흐름은 다음과 같다:

```text
사용자 질문
 → Multi-Query 생성 (질문 확장)
 → Metadata Filtering (조건 기반 필터링)
 → Hybrid Search (BM25 + Vector)
 → Re-ranking (정렬 최적화)
 → 최종 문서 선택
 → LLM 답변 생성
```

* **Multi-Query**: 하나의 질문을 여러 형태로 변환하여 다양한 표현의 문서를 검색할 수 있도록 함
* **Metadata Filtering**: 질문에서 특정 기업명 등을 추출하여 검색 범위를 제한
* **Hybrid Search**: 키워드 기반(BM25)과 의미 기반(Vector) 검색을 결합
* **Re-ranking**: 검색된 문서 중 질문과 가장 관련 있는 문서를 재정렬
* **Chunking 전략**: 공고 데이터를 구조적으로 나누어 필요한 정보만 효율적으로 검색

---

### 🔹 코드 실행 방법:

1. 벡터 DB 및 BM25 인덱스 생성
2. 사용자 질문 입력
3. Multi-query 생성
4. 필터 조건 추출 및 적용
5. Hybrid Search 수행
6. Re-ranking 후 상위 문서 선택
7. LLM을 통해 최종 답변 생성

## WHY (의사결정 기록)

### 1. **Q**: 왜 이 방식을 선택했는가?

**A**:
기존 RAG 방식은 단일 벡터 검색에 의존하기 때문에,

* 사용자 질문이 모호한 경우 검색이 잘 되지 않고
* 특정 기술 스택이나 기업 조건을 정확하게 반영하기 어려운 문제가 있었다.

이를 해결하기 위해

* **Multi-Query**를 통해 검색 범위를 확장하고
* **Hybrid Search**로 키워드와 의미 기반 검색을 결합했으며
* **Metadata Filtering**을 통해 불필요한 데이터를 제거하였다.

그 결과 검색 정확도와 응답 품질을 동시에 향상시킬 수 있었다.

---

### 2. **Q**: 다르게 구현한다면 어떻게 했을까?

**A**:
대안으로는

* **HYDE 방식**을 사용하여 가상의 답변을 생성한 뒤 검색하는 방법이나
* **GraphRAG**를 적용하여 문서 간 관계를 활용하는 방법이 있다.

하지만

* HYDE는 잘못된 정보 생성 위험이 있고
* GraphRAG는 구축 비용이 높아 현재 프로젝트 범위에서는 적합하지 않다고 판단하였다.

따라서 구현 난이도 대비 성능 향상이 확실한 방법을 우선적으로 선택하였다.

## 트러블슈팅 로그

| # | 문제 상황                        | 에러 메시지 | 원인 (Root Cause) | 해결 방법                      |
| - | ---------------------------- | ------ | --------------- | -------------------------- |
| 1 | 특정 회사 질문 시 다른 회사 데이터도 함께 출력됨 | 없음     | 필터링 없이 전체 검색 수행 | Metadata Filtering 추가로 해결  |
| 2 | 기술 스택 기반 검색 정확도 낮음           | 없음     | Vector 검색만 사용   | BM25를 결합한 Hybrid Search 적용 |
| 3 | 질문이 애매할 경우 검색 결과 부족          | 없음     | 단일 Query 사용     | Multi-Query 적용으로 개선        |
| 4 | 검색 결과는 맞지만 순서가 부정확           | 없음     | 유사도 기반 정렬 한계    | Re-ranking 추가              |
