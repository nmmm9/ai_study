RAG

Week 12

Agentic RAG.
검색하는 RAG에서
생각하는 RAG로.

2024년의 RAG는 한 번 검색하고 끝났다. 2026년의 RAG는 LLM이 검색을 직접 운영한다.

출발점

전통 RAG는 무엇을 할 수 없었나

한 번 검색하고 끝

임베딩, top-k 검색, 생성. 단방향 파이프라인이다. 검색이 빈약해도 그 결과로 답을 만들어야 한다.

단일 소스 한정

벡터 DB 하나만 본다. 내부 데이터와 외부 공시를 함께 봐야 풀리는 질문에 약하다.

Multi-hop 추론 부재

A를 찾고 그 결과로 B를 찾는 연쇄 추론이 한 번의 검색 안에 표현되지 않는다.

LLM이 수동 소비자

리트리버가 준 결과를 받아쓰는 역할에 그친다. 결과가 부족하다고 판단해도 다시 검색하지 않는다.

2026년 엔터프라이즈 AI 도입 보고서에 따르면, 2024년에 표준 RAG를 도입한 팀의 약 58%가 6개월 안에 다중 소스 다단계 추론을 한계로 지목했다.

PLAN

정의

Agentic RAG의 핵심

자율 에이전트가 중앙에 앉아 검색 전략을 계획하고, 적절한 도구로 라우팅하고, 자기 출력을 평가하고, 충분한 답이 나올 때까지 반복한다.

1

Plan

질문을 분해한다. 어떤 소스에서 무엇을 가져올지 먼저 결정한다.

2

Route & Retrieve

적절한 도구와 데이터 소스를 선택해 정보를 가져온다.

3

Evaluate & Loop

결과를 자기 평가한다. 부족하면 다시 검색하거나 다른 도구를 시도한다.

고정된 시퀀스가 아니라 조건부 그래프다. 이것이 2026년 엔터프라이즈 RAG의 표준 정의다.

Comparison

전통 RAG vs Agentic RAG

Traditional RAG

Linear Pipeline

  • 임베딩, top-k 검색, reranking, 생성을 1회 수행
  • 검색 실패 시 복구 경로가 없음
  • 출처는 벡터 DB 한 곳에 한정
  • LLM은 결과를 받아쓰는 역할
  • 구현이 단순하고 비용이 낮음

Agentic RAG

Dynamic Graph

  • Plan, Route, Retrieve, Evaluate를 반복
  • 실패 시 자동으로 다른 경로 시도
  • 벡터 DB, API, 도구 등 다중 소스 결합
  • LLM이 검색 전략을 직접 결정
  • 비용 3~10배, 응답시간 2~5배 증가

전통 RAG는 LLM을 결과 소비자로 봤다. Agentic RAG는 그 관계를 뒤집어 모델이 무엇을, 언제, 어떻게 가져올지 결정하게 한다.

Patterns

Agentic RAG의 다섯 가지 패턴

Router

질문 의도에 따라 소스로 분기한다. 가장 단순한 형태다.

ReAct

Thought, Action, Observation을 반복한다. 8주차에서 다룬 패턴이다.

Plan-Execute

계획을 먼저 세우고 단계별로 실행한다. 11주차의 핵심이다.

Multi-Agent

전문 에이전트가 분업한다. 9주차와 11주차에서 구현했다.

Self-RAG

검색 결과를 자기 채점한 뒤 부족하면 다시 검색한다. 12주차의 핵심이다.

실무에서는 하나의 패턴만으로 충분한 경우가 드물다. 12주차 프로젝트는 Router, Plan-Execute, Multi-Agent, Self-RAG 네 가지를 한 그래프 안에서 결합한다.

관점 전환

Retrieval의 범위가 넓어졌다

좁은 의미 (2020 교과서)

문서 임베딩 기반 벡터 검색.
top-k 결과와 reranking.
출처는 벡터 DB 한 곳.

넓은 의미 (2026 실무)

벡터 DB와 외부 API, 웹 검색의 통합.
SQL, GraphRAG, 메모리 조회까지 포함.
LLM이 어디서 어떻게 가져올지 결정.

Perplexity가 자신을 RAG 회사라 칭하는 이유, ChatGPT와 Claude가 웹 검색을 RAG라 부르는 이유는 동일하다. 출처가 무엇이든 LLM이 능동적으로 정보를 가져오면 RAG로 본다.

프로젝트 구조

두 종류의 Retrieval을 통합한다

Vector RAG

documents 도메인

사용자가 업로드한 PDF, TXT, MD 파일을 대상으로 한다.
ChromaDB에 영속 인덱스를 만들고, Self-RAG 패턴으로 검색 결과를 1~5점으로 자기 평가한다. 점수가 낮으면 쿼리를 재작성해 다시 검색한다.

API RAG

나머지 12개 도메인

외부 공공 API와 스크래핑 도구를 사용한다. 뉴스, 공시, 매물, 통계, 기상 등. Planner가 어떤 도구를 호출할지 결정하고, Replanner가 결과가 부족하면 도구를 다시 고른다.

본질은 같다. LLM이 답하기 전에 외부에서 정보를 가져온다. 출처가 벡터 DB인지 외부 API인지의 차이일 뿐이다. 두 가지 모두 RAG로 정의한다.

화살표(→)를 눌러 각 노드를 하나씩 살펴본다. 전체 그래프는 사용자 질문이 들어오는 START 부터 답변이 나가는 END 까지 일자형 흐름이고, documents 도메인과 Critic 에는 내부 루프가 들어 있다.

1 · START

사용자 질문이 그래프에 진입하는 지점. 이전 대화 기록(memory)과 함께 상태로 들어간다.

2 · Planner

질문을 1~6개의 step 으로 분해한다. 각 step 에 어떤 도메인이 필요한지 hint 를 붙인다. JSON 모드로 LLM 을 호출하기 때문에 결과가 정형화되어 다음 노드가 파싱하기 쉽다.

3 · Executor

plan 의 첫 번째 pending step 을 꺼내서 해당 도메인 에이전트로 디스패치한다. 한 step 씩 진행하기 때문에 결과가 다음 step 의 결정에 반영될 수 있다.

4 · 13 Domains

shopping, lifestyle, sports, news, finance, government, education, info, data, travel, culture, health, 그리고 documents. 각 도메인은 자기 도구 5~17개만 LLM 에 노출한다. documents 도메인 안에는 Self-RAG 루프(Query Rewrite → Search → Self-Eval → Re-search)가 따로 돈다.

5 · Replanner

step 결과를 보고 세 가지 중 하나를 결정한다. continue: 다음 step 으로 진행. revise: 남은 plan 을 새 step 으로 교체. finish: 충분하다고 판단하고 Writer 로 넘김. 최대 2회까지 plan 을 수정할 수 있다.

6 · Writer

모든 step 의 도구 결과를 모아 최종 답변을 작성한다. 스트리밍으로 토큰을 흘려보내고, documents 결과가 포함된 경우 인용 마커([1], [2])를 자동으로 부착한다. 재작성이 필요할 때는 Critic 의 피드백을 받아 두 번째 호출이 일어난다.

7 · Critic

Writer 의 답변을 1~10점으로 채점한다. 사실 정확성, 도구 결과 활용도, 답변 구조, 누락 네 기준이다. 7점 미만이면 issues 와 suggestions 를 Writer 에게 돌려보내 재작성 루프를 시작한다. 최대 2회 재작성 후에는 어떤 점수든 통과시킨다.

8 · END

최종 답변이 사용자에게 전달되고, 질문과 답변이 Checkpointer(thread_id 별 메모리)에 저장된다. 다음 대화에서 같은 thread_id 면 이 기록을 다시 불러온다.

Trade-off

비용은 늘지만 가치가 더 크다

방식토큰 비용응답 시간적합한 상황
Vanilla RAG1배1배단순 FAQ
Hybrid RAG1.5배1.2배일반적인 검색
Agentic RAG (단일 소스)3~5배2~3배다단계 추론이 필요한 질문
Agentic RAG (다중 소스)5~10배3~5배이질적 소스 간 비교, 검증

가치 있는 도메인

법률, 의료, 금융, 기업 지식 검색. 답변의 신뢰성이 비용보다 중요한 영역. multi-hop 분석이 필수인 질문.

오버킬인 경우

단순 FAQ, 단일 문서 Q&A, 한 번의 검색으로 답이 나오는 질문에는 기본 RAG가 더 적합하다.

Industry

실제 산업에서의 활용

헬스케어

환자 기록, 의학 논문, 약물 상호작용 DB를 결합해 진단을 권고한다. 다중 소스와 다단계 추론이 필요한 대표 영역이다.

법률 및 컴플라이언스

Harvey, Hebbia 같은 서비스가 판례, 계약, 규제, 사례를 통합 검색한다. 출처 인용이 필수다.

금융 분석

내부 데이터, 공시, 시장 데이터, 뉴스를 결합한다. 단일 소스로는 풀리지 않는 비교 분석이 일상적이다.

고객 지원

Sierra, Decagon 같은 서비스가 triage 에이전트를 두고 전문 에이전트로 라우팅한다.

리서치

Perplexity와 OpenAI Deep Research가 sub-question으로 분해하고 병렬 검색 후 합성한다.

개발 도구

Cursor의 Codebase 모드, Devin 같은 자율 SWE 에이전트가 코드 RAG와 실행 검증을 결합한다.

공통점은 두 가지다. 답을 자동으로 검증할 수 있다. 그리고 출처를 인용할 수 있다. 이 두 조건이 갖춰진 도메인에서 Agentic RAG의 ROI가 높다.

Research

2020년부터 2026년까지의 흐름

Lewis et al. (2020)

RAG라는 개념을 처음 제안한 논문. 단발 파이프라인 구조였다.

Asai et al. (2023) — Self-RAG

LLM이 reflection token을 통해 검색을 자기 평가하는 패턴. 12주차의 documents 도메인이 이 흐름을 따른다.

Chen et al. (2024) — CRAG

Corrective RAG. 검색이 실패하면 외부 웹 검색으로 자동 보완한다.

Edge et al. (2024) — GraphRAG

Microsoft가 발표한 패턴. 지식 그래프와 RAG를 결합해 entity 관계를 활용한다.

2026 — Agentic RAG 표준화

학계와 업계가 분류와 평가 기준을 정리하는 단계에 있다. SoK 논문이 등장했다.

2026 — Context Architecture

RAG가 Context Architecture라는 더 큰 추상화로 흡수되는 흐름이 시작됐다. 검색, 메모리, 도구의 통합이다.

Future

현재의 한계와 다음 단계

현재 한계

비용이 3~10배 증가한다.
응답 시간이 2~5배 길어진다.
디버깅이 어렵다. 어느 단계에서 틀렸는지 추적이 복잡하다.
평가 지표가 아직 미성숙하다. RAGAS, ARES 등이 발전 중이다.
다수의 LLM 호출 사이에서 일관성을 유지하기 어렵다.

다음 단계

Context Architecture: 검색, 메모리, 도구의 통합 추상화.
MCP 통합: 도구 표준화. Anthropic이 주도하고 있다.
Persistent Memory: 세션 간 학습.
Human-in-the-Loop: 민감한 결정에 사람 승인을 추가.
Specialized Critics: 사실, 스타일, 안전을 분리해 평가.

내 프로젝트가 이미 보유한 요소는 Checkpointer 메모리, Plan-Execute 그래프, Self-RAG 루프, 단일 Critic이다. 다음으로 추가할 만한 것은 Multi-Critic, MCP 통합, 영속 메모리다.

Wrap-up

정리

2026년 RAG의 정의

내 12주차 프로젝트

98개 도구를 통한 98가지 검색 방법.
Plan-Execute, Multi-Agent, Self-RAG, Critic의 결합.
Vector RAG와 API RAG의 통합.
한국 생활 도메인에서 2026 표준 패턴의 구현.

2020년의 RAG는 검색이 끝난 뒤에 LLM이 시작했다. 2026년의 RAG는 LLM이 시작하면서 검색을 끌고 다닌다. 12주차 K-Agent는 그 변화를 한국 생활 정보 도메인에서 그대로 구현한 사례다.
