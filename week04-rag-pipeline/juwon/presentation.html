06 / 12
Key Questions

이 3가지는 설명할 수 있어야 한다

Q1. 왜 Chunking을 해야 하나?

LLM에 넣을 수 있는 텍스트 길이(토큰)가 제한되어 있어서 문서 전체를 넣을 수 없다. 또한 문서 전체보다 관련 있는 부분만 잘라서 주는 게 더 정확한 답변을 유도한다.

Q2. 왜 Vector DB를 쓰나?

키워드 검색은 단어가 일치할 때만 찾는다. Vector DB는 의미(semantic)가 비슷한 문서를 찾는다. "자격요건"을 검색했을 때 "필수 역량"도 함께 찾아주는 게 그 예시다.

Q3. RAG가 Fine-tuning보다 좋은 이유는?

Fine-tuning은 새 데이터마다 모델을 다시 학습해야 해서 비용·시간이 크다. RAG는 문서만 교체하면 되어서 빠르고 저렴하다. 채용공고처럼 자주 바뀌는 데이터에 특히 적합하다.

07 / 12
Decision — Vector DB

FAISS 말고 ChromaDB를 선택한 이유

FAISS (이전 방식)

  • 메모리에만 존재
  • 프로그램 꺼지면 사라짐
  • 실행할 때마다 임베딩 재계산
  • 시간 & 비용 낭비
VS

ChromaDB (선택)

  • 디스크에 파일로 저장
  • 꺼도 데이터 유지
  • 다음 실행 시 바로 사용 가능
  • 실제 서비스처럼 동작
선택 이유

선배한테 "실제로 설치해서 파일로 저장되는 DB를 써보라"는 피드백을 받았다. 단순히 라이브러리로 불러오는 게 아니라, 실제 DB처럼 데이터가 남아 있는 게 더 현실적인 구현이라고 생각해서 ChromaDB를 선택했다.

Embedding

문장 → 384차원 벡터 → 유사도 검색

사용 모델

paraphrase-multilingual-MiniLM-L12-v2

한국어 포함 50개 언어 지원. 텍스트를 384개 숫자(벡터)로 변환한다.

임베딩 예시

"Java 개발 경험" → [0.21, −0.45, 0.88 ... ] (384개)
"백엔드 자격요건" → [0.19, −0.41, 0.77 ... ] (384개)

두 벡터가 가까울수록 = 의미가 비슷한 문장. 코사인 유사도로 측정.

검색 흐름

질문 → 같은 모델로 벡터 변환 → ChromaDB에서 코사인 유사도 top-10 검색 → Re-ranker로 top-3 최종 선정

Re-ranking Strategy

검색을 두 번 하는 이유

1단계 — ChromaDB (Bi-Encoder)

  • 질문 → 벡터로 변환해서 검색
  • 빠르지만 맥락 파악이 부족
  • top-10 후보 추출

2단계 — Cross-Encoder (Re-ranker)

  • 질문 + 청크를 함께 넣어서 관련성 계산
  • 느리지만 훨씬 정확
  • top-10 → top-3 정밀 선정
질문: "카카오 자격요건"
↓ 1단계 top-10
카카오 자격요건 · 카카오 우대사항
네이버 자격요건 · ... 10개
↓ 2단계 재정렬
✓ 카카오 자격요건 (0.94)
✓ 카카오 우대사항 (0.87)
✓ 카카오 주요업무 (0.81)
선택 이유

1단계만으로는 가끔 엉뚱한 회사 청크가 섞였다. Re-ranker를 추가하니 정확도가 눈에 띄게 향상됐다.

Troubleshooting

발견하고 고쳤던 3가지

📦 청크 크기가 너무 컸다
chunk_size=400 → 카카오 자격요건·우대사항이 다 합쳐서 청크 1개. 검색해도 정보를 찾지 못함.
→ chunk_size 400 → 200으로 변경. 51개 → 129개 청크. 섹션 단위로 분리됨.
🏢 "카카오 자격요건" 검색했는데 다른 회사가 나옴
자격요건 청크 안에 "카카오"라는 단어가 없음 → 임베딩이 어느 회사 청크인지 알 수 없음.
→ 저장할 때 "카카오 - 자격 요건: ..." 식으로 회사명을 청크 앞에 붙임. 해결.
🔒 ChromaDB 파일이 잠겨서 삭제가 안 됨
Streamlit 실행 중에 chroma.sqlite3 파일을 다른 프로세스가 점유.
→ 사실 삭제할 필요가 없었음. build() 함수 안에 delete_collection()이 있어서 재시작하면 자동 재구축.

느낀 점 & 다음에 해보고 싶은 것

💡 RAG의 핵심을 이해했다

LLM이 만능이 아니라는 걸 알게 됐다. 문서를 검색해서 LLM에게 "힌트"를 주는 방식이 얼마나 강력한지 직접 구현하면서 체감했다.

🔍 검색 품질이 전부다

RAG에서 가장 중요한 건 검색이었다. 청크 크기, 접두어 추가, Re-ranker까지 — 검색을 잘 해야 GPT가 좋은 답을 할 수 있다.

🗄️ 실제 DB를 써봤다

메모리에만 사는 FAISS가 아니라 파일로 저장되는 ChromaDB를 써봤다. 재시작해도 데이터가 남아 있는 게 실제 서비스와 같다는 걸 알았다.
