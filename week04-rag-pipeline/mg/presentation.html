PART 1

LLM이란?

Large Language Model = 대규모 언어 모델

핵심 원리

수조 개의 텍스트 데이터로 학습하여 "다음에 올 단어(토큰)를 예측"하는 모델

모델 규모

수십~수천억 개의 파라미터. GPT-4o는 약 200B+, GPT-5는 더 대규모 추정

할 수 있는 것

대화, 번역, 요약, 코딩, 분석, 창작 등 범용적 자연어 처리

GPT-4o / 5 (OpenAI) Claude 4 (Anthropic) Gemini 2.5 (Google) LLaMA 4 (Meta)

LLM 동작 원리

핵심은 "다음 토큰 예측" (Next Token Prediction)

1. 토큰화 (Tokenization)

오늘 날씨 가 좋 으니

텍스트를 토큰(단어 조각) 단위로 분리. 한글은 음절/형태소 단위로 분리됨.

2. 임베딩 (Embedding)

각 토큰을 고차원 벡터로 변환. "왕 - 남자 + 여자 = 여왕" 같은 의미 연산이 가능해짐.

3. Transformer 처리

Self-Attention으로 모든 토큰 간의 관계를 계산. "그것"이 무엇을 가리키는지 문맥에서 파악.

4. 다음 토큰 예측

# 입력
"오늘 날씨가 좋으니"

# 모델이 계산한 다음 토큰 확률
"산책"   → 32%
"외출"   → 24%
"소풍"   → 18%
"기분이" → 12%
 ...

# Temperature = 0 → 가장 높은 확률 선택
# Temperature = 1 → 확률에 따라 랜덤 선택

한 번에 한 토큰씩 생성하고, 그 결과를 다시 입력에 추가하여 반복하는 방식으로 문장을 완성

Pre-training

인터넷, 책, 논문 등 수조 토큰의 텍스트를 학습. "다음 단어 맞추기"만으로 언어의 패턴, 지식, 추론 능력을 습득.

수천 대의 GPU로 수개월 학습. 비용 수천만~수억 달러.

SFT (Supervised Fine-Tuning)

"질문 → 이런 식으로 답변하라"는 고품질 예시를 학습. 대화 형식을 익힘.

수만~수십만 개의 사람이 작성한 Q&A 데이터 사용.

RLHF / RLAIF

여러 답변 중 어느 것이 더 좋은지 사람(또는 AI)이 평가. 보상 모델로 강화학습.

안전성, 유용성, 정확성 기준으로 모델 행동을 조정.

PART 2

LLM은 만능이 아니다

학습 데이터 한계

학습 시점 이후의 정보를 알 수 없다. GPT-5도 학습 커트오프 이후의 법률 개정이나 사내 공지를 모름.

할루시네이션

모르는 내용도 그럴듯하게 지어낸다. GPT-5.4에서도 여전히 가짜 논문, 허위 통계를 생성할 수 있음.

도메인 지식 부족

내부 문서, 사내 매뉴얼, 비공개 데이터에 대한 지식이 전혀 없음.

이 세 가지 한계는 모두 같은 근본 원인 → LLM은 학습 데이터에만 의존한다
다음 토큰 예측이 아무리 뛰어나도, 학습하지 않은 정보는 생성할 수 없다

이 문제를 어떻게 해결할까?

방법 설명 단점
Fine-tuning 모델 자체를 재학습 비용 매우 높음, 업데이트마다 재학습 필요
Long Context에 직접 넣기 GPT-5의 128K 컨텍스트에 문서 전체 삽입 토큰 비용 폭증, 수십만 자 문서는 여전히 불가
RAG BEST 질문과 관련된 부분만 검색해서 전달 비용 효율적, 실시간 업데이트, 2026년 프로덕션 표준

RAG = 검색(Retrieval) + 생성(Generation)을 결합한 2026년 현재 가장 보편적인 프로덕션 AI 아키텍처

PART 3

RAG란 무엇인가?

Retrieval-Augmented Generation

사용자의 질문을 받으면,
1. 관련 문서를 검색하고
2. 검색 결과를 LLM에 함께 전달하여
3. 정확한 답변을 생성하는 기법

= "오픈북 시험"과 같은 원리.
LLM이 기억(파라미터)에만 의존하지 않고, 관련 자료를 참고하여 답변한다.

RAG의 전체 흐름

사전 준비 (Indexing)

1. 문서 수집
PDF, 웹페이지, 사내 문서 등
↓
2. 청킹 (Chunking)
긴 문서를 적절한 크기로 분할 2주차
↓
3. 임베딩 (Embedding)
텍스트 → 벡터로 변환 3주차
↓
4. Vector DB 저장
ChromaDB에 벡터 저장 3주차

질의응답 (Querying) 4주차

1. 질문 임베딩
질문을 같은 벡터 공간으로 변환
↓
2. 벡터 검색
코사인 유사도로 관련 청크 Top-K개 검색
↓
3. 컨텍스트 구성
검색된 청크를 프롬프트에 삽입
↓
4. LLM 생성
문서 기반의 정확한 답변 생성

전체 소요시간의 대부분은 LLM 생성 단계에서 발생

Step 1. 질문 임베딩

사용자의 질문을 벡터(숫자 배열)로 변환한다.

  • 모델: text-embedding-3-small (OpenAI)
  • 차원: 1536차원 벡터
  • 같은 의미의 텍스트는 비슷한 벡터를 가짐
  • 문서 청크와 같은 모델로 임베딩해야 함
# 질문 임베딩
"스타트업 초기 자금은?"

    ↓ OpenAI Embedding API

[0.012, -0.034, 0.056, ...]
 # 1536차원 벡터

# 소요시간: ~100ms
# 비용: ~$0.000002 ($0.02/1M tokens)

Step 2. 벡터 검색

질문 벡터와 가장 유사한 문서 청크를 찾는다.

  • 방법: 코사인 유사도 (Cosine Similarity)
  • 상위 K개 결과 반환 (기본 top_k=5)
  • ChromaDB가 내부적으로 최적화된 검색 수행
  • 키워드 검색이 아닌 의미 기반 검색

키워드 검색 vs 벡터 검색

키워드: "초기 자금" → "자금"이 포함된 문장만 검색
벡터: "초기 자금" → "시드 투자", "부트스트래핑 비용" 등 의미적으로 유사한 내용도 검색 

# 검색 결과 (Top-5)

[1] 유사도 92.3%  청크 #14
  "시드 단계 자금 조달은..."

[2] 유사도 88.7%  청크 #15
  "부트스트래핑으로 시작하는..."

[3] 유사도 85.1%  청크 #22
  "엔젤 투자자에게 접근..."

[4] 유사도 81.4%  청크 #3
  "초기 창업 비용 계획..."

[5] 유사도 78.9%  청크 #28
  "정부 지원금 활용..."

# 소요시간: ~5ms (매우 빠름!)

Step 3. LLM 생성

검색된 문서를 컨텍스트로 포함하여 LLM에 전달한다.

  • 시스템 프롬프트에 역할 + 규칙 설정
  • 검색된 청크를 번호와 함께 컨텍스트로 구성
  • "문서에 없는 내용은 없다고 답하라" 지시
  • 모델: gpt-4o-mini (기본) / gpt-4o
  • GPT-5.x도 사용 가능하나, 4o-mini가 RAG에 가성비 최고
# LLM에 보내는 프롬프트 구조

System:
 다음 문서를 참고하여 답변하세요.
 문서에 없는 내용은 '문서에 해당
 정보가 없습니다'라고 답하세요.

 ---
 [1] 시드 단계 자금 조달은...
 [2] 부트스트래핑으로 시작하는...
 [3] 엔젤 투자자에게 접근...
 ---

User:
 스타트업 초기 자금은?

Assistant:
 문서에 따르면, 스타트업 초기
 자금 조달 방법으로는...

핵심: 프롬프트 설계

RAG의 품질은 프롬프트 설계에 크게 좌우된다.

System Prompt (단일 Q&A)

다음 문서를 참고하여 질문에
답변하세요.

문서에 없는 내용은
'문서에 해당 정보가 없습니다'
라고 답하세요.

→ 할루시네이션 방지를 위해 "없으면 없다고 하라"는 지시가 핵심

System Prompt (챗봇)

당신은 문서 기반 질의응답
챗봇입니다.

아래 문서를 참고하여 대화에
답변하세요.

이전 대화 맥락을 고려하여
자연스럽게 답변하세요.

→ 멀티턴 대화에서는 이전 맥락 유지 지시가 추가

PART 5

RAG 챗봇

단일 Q&A를 넘어, 대화 흐름을 유지하는 멀티턴 챗봇으로 확장

단일 Q&A vs 챗봇

단일 Q&A 질문 1개 → 답변 1개
이전 대화를 기억하지 않음
RAG 챗봇 대화 히스토리 유지
"아까 말한 그 부분에서..." 같은 후속 질문 가능

대화 예시

스타트업 창업에 뭐가 필요해?
문서에 따르면, 사업 아이디어 검증, 팀 구성, 초기 자금...
그 중에서 자금은 어떻게 마련해? ← "그 중에서"를 이해
초기 자금 조달 방법으로는 부트스트래핑, 시드 투자...

챗봇 파이프라인 구조

질문
→
Embed
→
Search
→
Generate
+ 대화 히스토리
→
답변

단일 Q&A와의 차이점

  • LLM에 보내는 messages에 이전 대화 내용이 포함됨
  • 최근 10턴까지의 대화를 유지 (너무 길면 토큰 비용 증가)
  • 매 질문마다 새로 검색 — 대화 맥락이 달라지면 다른 청크가 검색됨

챗봇: 메시지 구조

# LLM에 보내는 전체 메시지 배열

messages = [
  {
    "role": "system",
    "content": """
      당신은 문서 기반 질의응답 챗봇입니다...
      ---
      [1] 시드 단계 자금 조달은...  ← 매번 새로 검색된 문서
      [2] 부트스트래핑으로 시작하는...
      ---
    """
  },
  {"role": "user", "content": "스타트업 창업에 뭐가 필요해?"},      ← 이전 대화
  {"role": "assistant", "content": "사업 아이디어 검증, 팀..."},  ← 이전 답변
  {"role": "user", "content": "그 중에서 자금은?"}               ← 현재 질문
]

데모 아키텍처

Frontend

Next.js + React + Tailwind CSS

  • 문서 선택 & 임베딩
  • 단일 Q&A 인터페이스
  • 파이프라인 단계 시각화
  • 챗봇 UI (버블 형태)

Backend

FastAPI (Python)

  • /api/embed — 문서 임베딩
  • /api/rag — 단일 Q&A
  • /api/chat — 챗봇
  • /api/collections — 컬렉션 관리

LLM GPT-4o-mini ($0.15/1M in) / GPT-4o ($2.50/1M in)

Vector DB ChromaDB 1.5.x (로컬 PersistentClient)
백엔드 FastAPI + Python 3.12
프론트 Next.js 16.1 (Turbopack) + React 19
스타일 Tailwind CSS v4

핵심 정리

LLM = 다음 토큰 예측

대규모 텍스트로 학습하여 다음 단어를 예측하는 모델. 강력하지만 학습 데이터 이후 정보는 모름.

RAG = 검색 + 생성

LLM이 외부 문서를 참고하여 답변하는 기법. 할루시네이션 방지와 최신 정보 활용의 핵심.

Pipeline: 3단계

질문 임베딩 → 벡터 검색 → LLM 생성. 전체 문서가 아닌 관련 부분만 전달하여 효율적.

RAG 챗봇

대화 히스토리를 LLM 메시지에 포함하여 멀티턴 대화 지원. 매 턴마다 새로 검색.
