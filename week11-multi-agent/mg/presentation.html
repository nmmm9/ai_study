WHAT

정의

멀티 에이전트란?

여러 LLM 이 각자 다른 역할 로 협업해서 하나의 문제를 푼다.
한 LLM 이 다 하던 걸 여러 명이 분업 하는 패턴.

단일 에이전트

한 LLM이 모든 도구·역할·검증을 다 함. 도구 5개 까지는 괜찮지만 그 이상부터 정확도가 급락한다.

멀티 에이전트

Supervisor 가 라우팅, 도메인 전문가들이 실행, Critic이 채점, Writer가 통합. 역할을 LLM 사이에 나눈다.

왜 나누는가

4가지 이유

1

컨텍스트 효율

한 LLM 에 도구 52개 description 다 넣으면 매번 토큰 폭발. 각 에이전트는 자기 도구 5~12개만 봄.

2

역할별 system prompt

쇼핑/스포츠/Critic 등 도메인 특화 지시를 따로. 한 prompt에 다 넣으면 충돌하거나 묻힌다.

3

병렬 실행

"날씨 + KBO 결과" → lifestyle과 sports 가 동시에 실행. 단일 에이전트는 순차로만 가능.

4

자기 검증 (객관성)

다른 에이전트가 답을 채점하면 자기 편향이 줄어든다. Critic 패턴의 핵심.

Patterns

주요 5가지 패턴

① Supervisor

1명이 라우팅 → N명이 실행. 가장 단순하고 흔한 패턴. 9주차.

② Hierarchical

Supervisor 가 또 Supervisor 를 가짐. 회사 조직처럼 다층 구조.

③ Self-Correction

평가자(Critic)가 답을 채점 → 재작성. 10주차.

④ Plan-and-Execute

계획 → 단계별 실행 → 재계획. 11주차 ★

⑤ Debate

여러 에이전트가 다른 시각으로 평가 → 토론 → 합의. Multi-Critic 변형.

패턴의 조합

실무는 보통 2~3개 패턴을 합쳐서 쓴다. 11주차는 Supervisor + Self-Correction + Plan-and-Execute 의 합성.

How they actually use it

실제 사용 방식 3가지

CASE A · Cursor / Devin

Plan + Self-Correction

1. Planner: 작업 분해
2. Editor: 코드 작성
3. Runner: 컴파일 / 테스트
4. Reviewer: 실패 시 재작성

검증 = 테스트 통과 여부

CASE B · Sierra / Decagon

Supervisor (Domain Routing)

1. Intent 분류 (Triage)
2. Billing / Tech / Sales agent 라우팅
3. 해결 못 하면 사람으로 escalation
4. 만족도 측정 → 학습

검증 = 사용자 해결 확인

CASE C · Deep Research

Plan-and-Execute

1. 질문 → 5~20개 sub-question 분해
2. 각 sub-question 병렬 검색
3. 결과 누적 → 추가 검색 필요시 revise
4. Synthesizer: 보고서 생성

검증 = 출처 매칭, 일관성

실무 공식: 패턴 1개로 충분한 경우는 거의 없다. Cursor = Plan-and-Execute + Self-Correction. Sierra = Supervisor + Critic. 2~3 개 패턴의 조합이 실제 production 의 기본 모양이다.
WHY

Why learn this

왜 알아야 하는가

1

AI 제품의 기본 모양이 멀티 에이전트로 바뀌고 있다

2023년: 단일 ChatGPT 인터페이스. 2025년: 거의 모든 AI 제품이 multi-agent (Cursor, Devin, Sierra, Deep Research, ChatGPT의 도구 모드까지). 단일 LLM API 호출만으로는 경쟁 못한다.

2

채용 시장의 핵심 스킬

AI Engineer 채용 공고의 80%+ 가 LangGraph / LangChain / AutoGen 경험을 요구. 단일 prompt engineering 만으로는 더 이상 차별화되지 않음.

3

단일 LLM 의 한계가 보인다

도구 5개 넘으면 정확도 급락, 도메인 지식 충돌, 자기 답변 못 검증, 복잡 질문 처리 한계. 구조적 한계는 구조적 해법으로 — 더 큰 모델이 아니라 더 나은 그래프.

4

다음 5년의 AI 가 갈 방향

단순 채팅 → 도구 사용 → 자율 에이전트 (multi-agent + 메모리 + 도구). Anthropic / OpenAI / Google 모두 같은 방향으로 투자 중. 멀티 에이전트는 그 토대.

한 줄 요약: 지금 단순 ChatGPT 래퍼만 만들 줄 알면 1~2년 안에 가치가 떨어진다. 멀티 에이전트를 설계할 줄 아는 것이 2025년 이후의 AI 개발자 기본기다.

PLAN

11주차 패턴

Plan-and-Execute

LLM 이 답변을 하기 전에 "내가 무엇을 어떤 순서로 할지" 를 먼저 적어둔다.
그 plan 자체가 그래프 상태에 들어가고, 실행 중에 다시 쓸 수 있다.
P

Plan

질문을 1~6개 step 으로 분해. 각 step 에 도메인 hint.

E

Execute

한 step 씩 진행. 해당 도메인 에이전트가 도구 호출.

R

Re-plan

매 step 결과를 보고 continue / revise / finish 결정.

Execute 루프 + Critic 루프 = Double Cycle. "무엇을 할지 (plan)" 와 "어떻게 답할지 (answer)" 를 각각 다른 루프로 개선한다.

예시

복합 질문 분해

"서울 날씨 + 어제 KBO 결과 + 강남구 맛집까지 한 번에"

1

lifestyle · 서울의 현재 날씨 정보 검색 → continue

2

sports · 어제 KBO 야구 경기 결과 검색 → continue

3

lifestyle · 강남구 맛집 검색 → 결과 빈약 → revise

4

lifestyle · (revised) 다른 도구로 강남 맛집 재시도 → finish

✓

Writer + Critic — 누적 결과로 답변 작성, 8/10 통과

한 문장이 4 step + critic 으로 분해됨. 각 step 후 Replanner 가 "계속 / 수정 / 종료" 를 결정.

Research Context

멀티 에이전트의 학술 흐름

Plan-and-Solve (Wang et al., 2023)

"먼저 계획, 다음에 실행" two-stage prompting. ReAct 대비 다단계 추론 개선.

BabyAGI / AutoGPT (2023)

자율 에이전트의 효시. plan-task queue-execution 사이클이 11주차의 직접 조상.

AutoGen (Microsoft, 2023)

Conversable agents 프레임워크. 에이전트 간 대화로 협업하는 표준.

MetaGPT (2023)

SOP (Standard Operating Procedure) 기반 멀티 에이전트. 회사 조직처럼 역할 분담.

CrewAI (2024)

역할 + 목표 + 도구로 에이전트 정의. 직관적인 협업 모델.

LangGraph (LangChain, 2024)

그래프 기반 에이전트 오케스트레이션. 우리 9~11주차의 토대.

When to use

멀티 에이전트가 빛나는 순간

✓ 좋을 때

· 도구가 10개 이상
· 도메인이 명확히 분리됨
· 답변 품질이 중요 (Critic 가치)
· 복합 질문이 자주 옴
· 다단계 조사가 필요

✗ 오버킬일 때

· 도구 3~4개로 충분
· 단일 도메인
· 응답 속도가 절대 우선
· 비용이 빠듯함
· 단순 FAQ 봇

핵심 관점: 멀티 에이전트는 "공짜로 똑똑해지는 마법" 이 아니다. 비용을 더 써서 정확도와 복잡 처리 능력을 사는 거래. 항상 단일 → 분리 → 다시 합치는 진화 과정을 거쳐야 한다.
